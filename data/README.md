# Data directory

Raw datasets live here but are **not** committed to Git — the Divar export is roughly 1 M rows × 64 columns. Versioning is delegated to DVC, which stores the file content out-of-band and commits only a small `.dvc` pointer.

## Expected layout

```text
data/
├── raw/
│   ├── Divar.csv                      # required for ML pipelines (DVC-tracked)
│   ├── Divar.csv.dvc                  # small pointer file — this is what Git tracks
│   ├── .gitignore                     # generated by `dvc add`, ignores Divar.csv
│   └── iran_city_classification.csv   # optional, used by statistical_analysis.ipynb
└── processed/                         # generated by DVC stages — do not edit by hand
    ├── price_train.parquet            # 80 % of sell listings (post feature engineering)
    ├── price_val.parquet              # 20 % validation
    ├── credit_train.parquet           # ~70.5 % of rent/credit listings
    ├── credit_val.parquet             # ~12.75 % validation
    └── credit_test.parquet            # 15 % held-out test (credit only — see project README for why)
```

`data/processed/` is a DVC output: deleting it is safe — the next `dvc repro` will regenerate it from the raw CSV.

## Why DVC, not Git LFS or a download script

- **Reproducibility.** The DVC pipeline tracks data dependencies the same way it tracks code; a feature-engineering change only re-runs the affected stages.
- **Small Git history.** Only `.dvc` pointers and code go into Git, so clones stay fast.
- **Remote-agnostic.** Add an S3, GCS, or SSH remote with one command (`dvc remote add`) when the team is ready to share data; until then, DVC works entirely on the local filesystem.

## Setup

### 1. Obtain `Divar.csv`

Download from your data source and place at `data/raw/Divar.csv`. If you keep the file elsewhere, set `DIVAR_CSV=/absolute/path/to/Divar.csv` in `.env` instead of copying it.

### 2. Track it with DVC (first time only)

```bash
pip install -e ".[mlops]"
dvc add data/raw/Divar.csv
git add data/raw/Divar.csv.dvc data/raw/.gitignore
git commit -m "Track Divar.csv with DVC"
```

### 3. Run the pipeline

```bash
dvc repro train_price    # validate → prepare → validate → train → drift report
dvc repro train_credit
```

`dvc repro` is idempotent — it re-runs only stages whose inputs (code or data) have changed.

## Without DVC

For a quick local experiment without DVC tracking, the CLIs work directly off the raw CSV:

```bash
pip install -e .
divar-prepare-price
divar-train-price --from-processed --mlflow
```

You give up reproducibility guarantees and stage caching, but it is useful for one-off exploration.

## Schema expectations

`src/divar/validation/schemas.py` declares the **raw** and **processed** schemas in Pandera. The `validate_raw` stage runs before any feature engineering and will fail loudly with a row count and a list of offending columns if the raw CSV drifts. Update the schema there whenever upstream Divar changes a column name or dtype.
